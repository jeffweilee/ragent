| 2026-05-03 | YAGNI / Plugin | Risk of premature abstraction in `ExtractorPlugin` Protocol if shaped around hypothetical Phase 3 graph needs. | Designing for unverified future requirements violates YAGNI. | **[Rule]** Plugin Protocol fields must be justified by an existing in-Phase consumer. | 2026-05-04 | Spec-Rule sync | Spec §7.1 contained DB FK + JSON ACL column, in conflict with `00_rule.md` (no physical FK; OpenFGA is ACL system of record). | Spec was authored before reading the latest `00_rule.md`. | **[Rule]** Whenever `00_rule.md` is updated, the next agent-team round must re-validate spec/plan against the new rule version and cite it in the team output. |
| 2026-05-04 | Performance | Spec said `{ESVector ∥ ESBM25}` and used parallel-bar notation, but Haystack sync Pipeline runs components sequentially — actual chat latency was additive, not the implied max. Embedder catalog described retries but no batching, so a 500-chunk doc would issue 500 HTTP calls. | Notation borrowed from a parallel-runtime spec was applied to a sequential runtime; batching defaulted to 1 because the contract did not specify. | **[Rule]** Concurrency notation (`∥`, `par`, "concurrent", "fan-out") in the spec must be matched by a **runtime that actually runs in parallel** in that phase, or annotated `(sequential in PX)`. Every call to a remote service that accepts arrays must declare an explicit batch size in the catalog (`batch=N`). Per-call timeouts are mandatory in §4.3 / §4.5; "implicit default" is not acceptable. |
| 2026-05-04 | Retrieval/ACL | Earlier round 3 declared "OpenFGA out-of-scope across all phases" and proposed an owner-based ES filter (`terms(owner_user_id ∈ [user_id])`) as the P2 ACL. That decision is **reversed** in B14: the owner-on-ES approach couples auth to the retrieval index (re-index on every permission-model change), forces a single-relation model (no sharing/groups/roles), and overloads the column name `owner_user_id` with authorization semantics it was never going to keep. | Auth-shape decisions were taken under "minimum-viable for P2" pressure without weighing the lifetime cost of putting auth fields into the retrieval index. The "no out-of-band ACL service" framing was treated as a hard constraint when the real constraint was "no per-query out-of-band call inside the retrieval hot path" — which a post-retrieval gate satisfies. | **[Rule]** Auth/permission fields **MUST NOT** live on the retrieval index (ES, vector store, graph store). Permission gating is a **post-retrieval layer** behind a Protocol; the backend (OpenFGA, opaque DB lookup, etc.) is swappable without re-indexing. Column names in the data layer must describe **what they record** (`create_user` = creator audit), never **what they authorize** (`owner_user_id` implied an ACL role the column did not enforce). When reversing a prior Decision-Log row, append a new dated row that explicitly cites the superseded ID — never edit the old row in place. |
| 2026-05-04 | Interface freeze | B15 added "embed `f"{title}\n\n{chunk_text}"`" to `VectorExtractor.extract` after the Plugin Protocol was frozen — but the Protocol signature is `extract(document_id) -> None` with no way to pass `source_title`. Caught only at Round-4 audit; would have produced an `AttributeError` deep in T1.12 Green phase, blocking the track until a Protocol-or-DI decision was forced. | Adding a new data dependency to a method whose interface is frozen happened without checking the existing read path. The frozen-Protocol invariant was respected; the consumer-side audit was not performed. | **[Rule]** When a new requirement introduces data a frozen interface cannot pass, the **default resolution is constructor injection in the implementation, not Protocol widening**. Before merging the new requirement, audit every consumer of the data: does each have a read path? If not, add the constructor dependency. Audit checklist for any cross-cutting field addition: list every consumer, mark `direct-arg` / `injected-dep` / `missing`. A `missing` blocks the change. |
| 2026-05-04 | Capacity planning | B2 capped CSV at 50 MB ("no row cap, inherit file-size limit") with no worst-case math. Round-4 audit ran the multiplication: 50 MB × 50 chars/row × 1 sentence/row × 30 s/embed ≈ 250 hours. The "no cap" decision was implicitly assuming small CSVs; large CSVs broke the system's wall-clock budget silently. B24 (RowMerger) restored the bound to ~25 k chunks. | "No cap" decisions are easy to make and hard to verify. The cost surfaces as a worst-case scenario the spec author didn't compute. | **[Rule]** Every spec decision of the form "no cap on N" / "bounded only by X" **MUST include the worst-case multiplication** in the same paragraph: `worst_case = max_X × cost_per_unit`. If the result exceeds an existing system budget (timeout, memory, count, cost), the "no cap" is implicit and false — introduce packing, aggregation, or a hard cap before the decision is accepted. Applies to file-size limits, list endpoints, fan-out plugins, retry counts. |
| 2026-05-04 | DB design | `documents.storage_uri` stored full URIs like `minio://ragent-staging/<key>` even though the bucket name is constant per deployment. Renaming the bucket would require an `UPDATE documents SET storage_uri=...` migration. B25 split the data into `object_key` (per-row) + `MINIO_BUCKET` (env-driven config). | Storing immutable per-deployment values inside per-row data conflates two lifecycles (config rotation vs row mutation). Defaulted because "URI is one string" felt simpler than two columns of split data. | **[Rule]** Columns storing values that are **constant for the deployment** (bucket name, region, hostname, default queue, cluster id) belong in **config**, not in rows. Litmus test: "if I rename this constant tomorrow, do I need to `UPDATE` every existing row?" If yes, the value is misplaced. Apply at schema-design time; once a row contains it, the migration cost compounds with the table size. |
| 2026-05-04 | ES Field Classes | Adding chat filter fields (`source_app`, `source_workspace`) re-opened the "should this go on the ES index?" debate that B14 (`owner_user_id` reversal) had previously settled with a "no". Risk: applying B14 too broadly would have forced an over-fetch+JOIN post-retrieval filter, silently truncating narrow scopes. Reality: B14 was specifically about **authorization** fields; **scope/content metadata** that already drives row identity (like `lang`, `title`) belongs on the index. | The lesson from B14 was over-generalised in conversational shorthand to "no business fields on ES". The actual rule is narrower — auth fields specifically. Without a written distinction, the next person making the call has to re-derive it under deadline pressure. | **[Rule]** Distinguish two field classes when deciding whether to denormalise onto a retrieval index: **(1) Authorization fields** (answer "can this caller see this row?" — owner, viewer set, ACL relation): NEVER on the retrieval index, regardless of phase. Use a post-retrieval Permission Layer Protocol. **(2) Scope/content metadata** (answer "what kind of content is this?" — language, source app, workspace, content type, title): denormalise freely when the access pattern requires filtering at scoring time. Litmus test: "if this field's value changes for a row, is it still fundamentally the same record?" Yes ⇒ scope metadata, denormalise. No (the row's identity changes) ⇒ likely auth-adjacent, keep off the index. Document the classification in the spec when the field is added; a Decision-Log row must explicitly state which class it belongs to. |
| 2026-05-04 | Typed dispatch | Initial `_fan_out_plugins` used `method: str` + `getattr(p, method)` to switch between `extract` and `delete`. This is a stringly-typed two-branch dispatch: a typo compiles fine and surfaces only at runtime as `AttributeError`; static analysis and IDEs cannot trace the call. Caught in `/simplify` review, not in tests. | "Pass a string and getattr it" felt shorter than "pass a callable" when drafting the initial implementation. The Protocol was already imported for type hints but not used to define the callable type. | **[Rule]** Two-branch method selection (`"extract"` vs `"delete"`, `"read"` vs `"write"`) MUST use typed `Callable` parameters, not `str` + `getattr`. The caller already holds the plugin reference; extract the callable at the call site (`p.extract`, `p.delete`) and pass it down. `getattr(obj, str)` is acceptable only for user-supplied dynamic attribute names (e.g. config-driven field names); never for internal dispatch between a known-finite set of methods. |
| 2026-05-04 | Bootstrap | `init_schema.py` split schema.sql by `;` and checked `if stmt.startswith("--")` to skip comments. The first statement chunk included the header comment block AND `CREATE TABLE documents` — starting with `--`, so the whole chunk was filtered. Only the second table (chunks) was created. Silent data-only write caused document-table insertions to fail at the first integration run. | SQL files often open with multi-line comment headers that precede the first DDL statement. Treating the comment prefix as a proxy for "skip this statement" ignores that real statements can follow on subsequent lines within the same split fragment. | **[Rule]** SQL statement extraction from `.sql` files MUST strip comment lines line-by-line before checking if the remaining fragment is non-empty — never apply `startswith("--")` to the whole fragment. Use a helper like `_strip_comments(raw)` that filters each line independently. Apply the same pattern to any code that parses SQL files (migration runners, schema-snapshot appliers, test fixtures). |
| 2026-05-04 | FastAPI | FastAPI parameters declared as bare default values using `File(...)`, `Form(...)`, `Header(...)` trigger ruff B008 ("Do not perform function call in default argument"). The failure appeared only at lint time, blocking CI while tests had already passed. Switching to `Annotated[Type, Dependency()]` = None resolved both the lint violation and improves FastAPI's OpenAPI schema generation. | FastAPI's older documented style used bare defaults; ruff B008 was added to catch exactly this class of bug (default evaluated once at import, not per call). The team copied the older style without checking ruff's rules. | **[Rule]** All FastAPI parameter declarations MUST use the `Annotated[Type, Dependency()]` pattern (not bare default `= Dependency()`). This applies to `File`, `Form`, `Header`, `Query`, `Body`, `Depends`, and any FastAPI-injected type. Catch this in the Red phase: include `ruff check .` as part of the test-pass criteria — a lint failure is a Red failure, not a post-Green cleanup. |
| 2026-05-04 | SQLAlchemy | `composition.py` used `with engine.connect() as conn:` and passed `conn` to repository constructors. `with` closes the connection at block exit, leaving repositories holding a closed connection that raises `InvalidRequestError` on the first query. The naive "fix" — hoist out of the `with` and hold one app-lifetime `Connection` — would have violated `00_rule.md` §Database Practices (Mandatory Connection Pool) and reintroduced the "Packet sequence error" / "command out of sync" failure mode under concurrent FastAPI traffic. | Two compounding mistakes: (1) the `with` context manager was applied to a resource passed into a longer-lived owner; (2) the dependency direction was wrong — singletons should not own a `Connection` at all. SQLAlchemy's context manager silently invalidates the connection on exit rather than raising at assignment time, masking the lifetime mismatch. | **[Rule]** Singletons (repositories, services, the composition root) MUST hold the `Engine`, **never** a `Connection`. Connections are checked out per unit of work via `with engine.begin() as conn:` (write) or `with engine.connect() as conn:` (read) inside each repository method, or injected per-request through a `Depends`-driven session factory. This satisfies `00_rule.md` §Database Practices and the present concurrency-safety requirement in one move. Forbidden patterns: `self._conn = engine.connect()` at `__init__`; passing a `Connection` (rather than an `Engine`) into a constructor; any module-scope `connection = engine.connect()`. Use `with engine.connect() as probe: probe.execute(text("SELECT 1"))` for one-shot startup health checks only. |
| 2026-05-04 | DI | `app.py` initially passed `broker=None` to `IngestService`. `IngestService.create()` unconditionally calls `self._broker.enqueue(...)`, so the first `/ingest` POST would have raised `AttributeError: 'NoneType' object has no attribute 'enqueue'`. The bug was caught by `/simplify` code review, not by any test, because the integration test that exercises the full create path requires Docker. | A `None` default was used as a "lazy placeholder" during initial wiring; the callee never guards against `None`. Without an integration test covering the real wiring, this class of error is invisible to the unit test suite. | **[Rule]** Never pass `None` for a required DI dependency unless the callee explicitly handles `None` (e.g. optional feature). When a constructor argument is typed non-Optional, pass the real instance. If the real instance requires env vars not available in test, use a mock in tests — do not default the production path to `None`. Review every `SomeClass(dep=None)` call site before committing; if the callee calls `dep.method()` unconditionally, `None` is a latent `AttributeError`. |
| 2026-05-04 | Container design | Initial `Container` exposed `_chat_router_kwargs: dict` (a private dict embedding `rate_limit` and `rate_limit_window`) accessed via `**container._chat_router_kwargs` in `app.py`. This leaked implementation details, bypassed type checking on the dict values, and made IDEs unable to trace the field names. | The dict was added as a convenience shortcut to pass multiple config values at once, but the Container is a transparent data object — it should expose named fields, not opaque dicts. | **[Rule]** `Container` (composition root) dataclass fields MUST be typed explicitly. Never use `dict` or `Any` fields as bundled config bundles — each distinct value gets its own named field. Private (`_`) fields on a dataclass that leak to callers are a design smell; if a caller needs the value, make it a public field with a precise type. |
| 2026-05-04 | Dependency management | `fakeredis` was initially placed in `[project].dependencies` in `pyproject.toml`, making it a runtime production dependency. It belongs in `[dependency-groups].dev`. A test-double in production dependencies inflates the production image, risks version conflicts, and signals to downstream users that a fake Redis is a supported dependency. | The package was added quickly during the Red phase without checking which section of `pyproject.toml` applies to test-only code. | **[Rule]** Test doubles, mock libraries, and test data generators (`fakeredis`, `pytest`, `pytest-cov`, `factory-boy`, etc.) MUST be placed in `[dependency-groups].dev` (or `[project.optional-dependencies].dev`), never in `[project].dependencies`. Before adding any package: ask "is this required to run the production binary?" If no, it belongs in dev. Audit `[project].dependencies` at the end of each track's Green phase for test-only packages. |
| 2026-05-05 | RAG | `/chat` and `/chat/stream` retrieved documents but never injected them into the LLM prompt; `sources[]` was decorative and answers were ungrounded despite passing T3.9/T3.11. | Spec §3.4 specified retrieval and `sources[]` shape but was silent on prompt construction; no test asserted on the `messages` argument passed to `llm_client.chat`, so the gap survived TDD. | **[Rule]** Every retrieval-augmented endpoint must have a Red test that asserts on the exact `messages` payload sent to the LLM (system + user wrapping), not just the response shape. | 2026-05-07 | Pipeline / cleanup | C6 dropped `_CharBudgetChunker`, `langdetect`, `nltk`, `ChunkRepository`, the `chunks` DB table, and the `_IdempotencyClean` step. The v2 pipeline (`_TextLoader → _MimeAwareSplitter → _BudgetChunker → DocumentEmbedder → DocumentWriter(ES)`) carries everything inside Document meta and the ES `chunks_v1` index — the v1 dual-store (chunks DB + ES) was dead weight from C2 onward but kept until the v2 pipeline was live. Two non-obvious traps surfaced: (1) Haystack's stock `FileTypeRouter` routes `ByteStream`/`Path`, not `Document`, so a custom `_MimeAwareSplitter` was needed to dispatch on `meta["mime_type"]`; (2) the e2e first run failed at the writer with `KeyError: 'create'` because `haystack-elasticsearch` reads `e["create"]` from bulk-error items even when policy is `OVERWRITE` (which sends `index` op), and ES rejected an all-zero embedding from the WireMock stub with a `magnitude_zero` error. | The cleanup was deferred behind C4 because removing the v1 chunker without a working v2 chunker would have left the pipeline unrunnable. Two staging mistakes: zero-embedding stubs are a foot-gun whenever the index uses cosine `dense_vector`; and the worker's `Pipeline.run` input shape changed from `{"converter":{"sources":[...]}}` to `{"loader":{"content":..,"mime_type":..}}` — every external test that reused the v1 input shape silently 415'd or FAILED. | **[Rule]** Drop a deprecated component only after its replacement is exercised end-to-end (unit + integration green) — never both at once. **[Rule]** WireMock embedding stubs MUST return non-zero vectors (`[0.01]*1024`, not `[0.0]*1024`); ES `dense_vector` cosine indices reject magnitude-zero with no easy diagnostic. **[Rule]** When a pipeline's run-input shape changes, audit every call site (worker module, integration tests, e2e tests) in the same commit; a 415/422 from `/ingest` is the symptom of a producer that never updated. **[Rule]** Custom `@component` classes wrapping mime-routing are preferable to forcing `FileTypeRouter` over `Document` inputs — Haystack's stock router signature is `ByteStream`/`Path` only and adapter shims add more code than the custom component they replace. |
| 2026-05-09 | Observability contract | Audit of the ingest→search→chat flow surfaced two coupled gaps: (a) **distributed seams emit no logs** — `ingest.dispatched` (producer-side after `kiq`), `ingest.task.started` (worker-side pickup), `ingest.deleted` (soft-delete service), `supersede.completed` (task tail), and the chat hydrator's `status='READY'` filter (silent drop) all had no business event, so an operator could not grep "did the message land?", "did the worker pick it up?", or "how many chunks were silently filtered out by the READY gate?"; (b) **the global `Exception` handler collapsed every typed domain exception into `error_code="INTERNAL_ERROR"`** — `IngestStepError(error_code="EMBEDDER_ERROR")` and any future upstream-service exception lost their domain code on the way out; the response body said "internal error" while the only place that knew it was an embedder failure was a worker log nobody was reading. Compounded: `GET /ingest/{document_id}` returned `{"status": "FAILED"}` with no `error_code` / `error_reason`, so a downstream API could not branch its retry policy on the actual failure mode. | Logs were added at the *component* boundary (per-step `ingest.step.*`) but not at the *service / task / seam* boundary; the global handler was written when no typed domain exceptions existed and was never re-examined when `IngestStepError.error_code` shipped; upstream client retries (`embedding`, `llm`, `rerank`) raise raw `httpx.HTTPStatusError` / `TimeoutException` which the handler cannot distinguish from a programming bug, so all three collapse to 500. | **[Rule]** Every domain-service operation in/out **must** emit a structured business log (entry + exit, success or failure with `error_code`); distributed seams require paired producer- and consumer-side logs; silent filters (`_SourceHydrator`, dedupe stages, ACL post-filters) **must** log a `dropped_count`. Recorded as `00_rule.md §Service Boundary Logs`. **[Rule]** Domain exceptions surfaced to callers **must** carry `error_code: str` + `http_status: int`; the global handler **must** read both via `getattr` and fall back to `INTERNAL_ERROR` only when `error_code is None`; upstream-service failures (retry exhaustion, timeout) **must** raise typed `UpstreamServiceError` (502) / `UpstreamTimeoutError` (504), never collapse to 500; async task failures **must** persist `error_code` + `error_reason` on the document row and the `GET /<resource>/{id}` endpoint **must** return them. Recorded as `00_rule.md §API Error Honesty`. Both rules carry mandatory paired test verification — an entry/exit log unit test for every new service method and a paired log/response test for every new domain exception. |
| 2026-05-06 | TaskIQ producer | `IngestService.create` and `Reconciler._redispatch_*` called `self._broker.enqueue(label, **kwargs)` on what was wired in production as a raw `taskiq_redis.ListQueueBroker`. `AsyncBroker` exposes **no** `.enqueue()` — the producer-side API is `await registered_task.kiq(**kwargs)` on the decorated task object. Additionally the producer process never called `await broker.startup()`, so even after fixing the call, the first `kiq` would fail with no Redis connection. Tests passed for months because `MagicMock` accepts any attribute, hiding the contract drift. A third defect: `reconciler._repair_multi_ready` enqueued `"ingest.supersede"` but no `@broker.task("ingest.supersede")` was registered (only the service method existed). | The `enqueue(label, **kwargs)` shape was an internal abstraction sketched at plan time and never backed by a real implementation; production wiring substituted the raw broker without anyone noticing. The producer-side `startup()` requirement of TaskIQ was not represented in any spec row or lifespan code. Mock-based unit tests cannot catch this class of contract drift. | **[Rule]** TaskIQ producer-side code MUST: (a) `await broker.startup()` once at process boot (FastAPI lifespan, reconciler `__main__`) and `await broker.shutdown()` at exit — failure here aborts boot, never silently degrades; (b) enqueue via the **decorated task object** (`await ingest_pipeline_task.kiq(**kwargs)`) or via a thin dispatcher that resolves the label through `broker.find_task(label)` and raises `RuntimeError` on miss — never call a method (`broker.enqueue`) that does not exist on `AsyncBroker`; (c) every task label string referenced by a producer MUST have a matching `@broker.task("<label>")` registration in a module that is imported by the producer process before first dispatch (asserted by importing `ragent.workers.<module>` in both `bootstrap/app.py` and `reconciler._build_from_env`); (d) at least one **integration test against `taskiq.brokers.inmemory_broker.InMemoryBroker`** must exercise the dispatcher contract end-to-end so mock-based unit tests cannot hide drift. Sync-from-async bridge: when a sync call site (FastAPI `run_in_threadpool` worker thread) must enqueue, wrap the async dispatcher in a sync facade that uses `anyio.from_thread.run` — works because `run_in_threadpool` ⇒ `anyio.to_thread.run_sync` topology — and document this constraint at the dispatcher class. |

## SRE

| Date | Topic | Description | Root Cause | Actionable Guideline |
| :--- | :--- | :--- | :--- | :--- |
| 2026-05-04 | Concurrency | Concurrent worker + Reconciler can both update `documents.status`/`attempt`, causing double-dispatch. | No explicit locking on state-machine mutations. | **[Rule]** All `documents.status` mutations use `SELECT … FOR UPDATE` (Worker) or `FOR UPDATE SKIP LOCKED` (Reconciler). Repository.`update_status` raises `IllegalStateTransition`. |
| 2026-05-04 | Cleanup Ordering | Worker deleted MinIO staging object **before** committing terminal `status=READY`, so a crash in the gap caused the next reconciler retry to 404 on MinIO and eventually mark the document `FAILED` despite a successful pipeline run. Mirror risk: supersede ordered by `created_at < self` was not order-safe — a newer doc finishing READY before an older doc left the older one orphaned. | Side-effect cleanup (and stateful comparisons) ordered before / against durable state commits. | **[Rule]** Within any worker, **durable state commits MUST precede external side-effect cleanup**: (1) commit terminal status; (2) best-effort cleanup of staging artefacts; (3) log orphan-tolerated event. For convergence rules across multiple writers (e.g. supersede), use a **set-based survivor selection** (`MAX`/`MIN` over the candidate set under `FOR UPDATE SKIP LOCKED`) rather than a self-referential pairwise comparison. Applies to MinIO, Redis staging, and any future transient store. |
| 2026-05-04 | Lock Scope | Long-running pipeline held `SELECT … FOR UPDATE` on the document row across the entire 60+ s pipeline body (embedder, ES, plugins). A live-but-hung worker would never release the lock, and the Reconciler's `SKIP LOCKED` would skip the row forever — silent stuck-PENDING. Worker process *death* releases the lock (DB drops session); worker *hang* does not. | Single transaction wrapped both the cheap status mutation and the expensive external pipeline body. | **[Rule]** Status mutations and external work MUST use **separate short transactions**. Acquire `FOR UPDATE NOWAIT`, write the new status, **commit**, run external work outside any DB tx, then re-acquire briefly to commit the terminal status. Never hold a row lock across an external call. Worker uses `NOWAIT` (re-kiq self on contention); only the Reconciler uses `SKIP LOCKED`. |
| 2026-05-04 | Reconciler Coverage | Recovery surface was incomplete: Reconciler swept only `PENDING/DELETING > 5 min`. UPLOADED orphans (TaskIQ message lost), multi-READY rows (supersede task lost between `status=READY` commit and kiq), and a missing reconciler heartbeat were all silent invariant violations with no automated repair. | New transient states / convergence rules added without extending the Reconciler in lockstep. | **[Rule]** Every status that is **not terminal** and every **eventual-consistency invariant** must have a Reconciler arm. When introducing a new transient state (e.g. UPLOADED, DELETING) or an async convergence task (e.g. supersede), add: (a) a sweep query keyed on age or invariant violation, (b) a BDD scenario, (c) a plan row. Reconciler itself emits a `reconciler_tick_total` heartbeat with a Prometheus alert at > 10 min absence (no watcher of the watcher). |
| 2026-05-04 | Heartbeat Liveness | Round-4 audit caught a no-lock-window race: §3.1's earlier reform (long-tx → two short txs, no lock during pipeline body) freed the row for `SKIP LOCKED` correctly, but inadvertently removed the implicit liveness signal that the held lock had provided. The Reconciler's "PENDING > 5 min" threshold could now pick up a still-running worker's row (no lock, no contention on `NOWAIT` reacquire) and re-dispatch it — silent double-processing. | When decomposing a long-held lock into short transactions, the lock's *secondary role* (implicit liveness signal to outside watchers) was not migrated. Spec captured the liveness concern only inside the lock contract; once the lock left the picture, the concern had no home. | **[Rule]** Whenever a long-held lock is decomposed into short transactions, **explicitly replace the lock-as-liveness signal** with a heartbeat (periodic timestamp update on the same row, default 30 s), a lease column, or an external watchdog. Reconciler / sweeper queries must key on the heartbeat freshness, never on raw status-age. New transient states added later inherit the same rule. Audit: every `FOR UPDATE` removal must answer "what now tells observers this row is being worked on?" |
| 2026-05-04 | Aggregate Timeout | Spec carried explicit per-call timeouts (Embedder 30 s/batch, ES bulk 60 s, plugin 60 s) but no aggregate ceiling around the pipeline body. A 50 MB CSV with 1 M short rows would issue ~31 k embedder batches × 30 s = 250+ hours of legal per-call time before any failure surfaced. T7.2 (100-doc 99% success rate) would have silently broken on first large CSV. Per-call protection is not loop protection. | The "every external call has a timeout" rule was applied successfully but stopped at the call boundary. Aggregate wall-clock budget for the *containing operation* was treated as implicit (= sum of declared per-call budgets), which is not the same as enforced. | **[Rule]** Per-call timeouts protect individual calls; they do **not** bound aggregates. Every loop, fan-out, batch sequence, or pipeline that issues N external calls **MUST carry an explicit aggregate wall-clock ceiling** (e.g. `PIPELINE_TIMEOUT_SECONDS`, `FAN_OUT_TIMEOUT_SECONDS`). Overrun maps to a deterministic terminal failure with a dedicated `error_code`. Audit checklist: for each spec call site, list (per-call timeout, aggregate ceiling). A blank in column 2 is a bug. |
| 2026-05-04 | Integration testing | Integration tests requiring live MariaDB, Elasticsearch, Redis, and MinIO were written without a Docker-availability guard. In environments where the Docker daemon is absent (local CI, GitHub Actions without Docker-in-Docker), every integration test fails with a socket-connection error rather than a clean skip. | Testcontainers-based tests assumed Docker daemon availability implicitly. There was no `pytest.mark.skipif` or conftest-level guard to detect and communicate the missing prerequisite. | **[Rule]** Every integration test file that uses Docker/testcontainers MUST apply `pytestmark = pytest.mark.docker`.
